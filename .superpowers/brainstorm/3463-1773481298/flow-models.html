Shore Excursion Booking Flow

Which booking journey model fits cruise passengers best?

A

Streamlined 4-Step Wizard

Ports → Activities → Schedule → Review

Remove Transport & Accommodation steps entirely. Keep the linear wizard but make it cruise-focused. User adds ports with dock times, picks activities, AI schedules within dock windows, then submits quote.

Pros

  • Familiar wizard pattern
  • Minimal code changes (just skip 2 steps)
  • Clear progression

Cons

  • Activities still scoped to one destination
  • Must complete all steps linearly
  • Scheduling feels separate from browsing
B

Port-First Explorer

Itinerary → Per-Port Activity Browsing → AI Auto-Schedule → Review

After entering ports, the user explores activities per port (tab/accordion for each port). AI auto-schedules as activities are added (no separate Schedule step). Review shows the full multi-port plan.

Pros

  • Activities tied to specific ports (natural for cruise)
  • No separate scheduling step — AI works in background
  • Feels like building a trip, not filling a form

Cons

  • Bigger UI refactor (merge Activities + Schedule)
  • Per-port tabs add complexity
  • Need activity catalog per destination
C

AI-First Concierge

Tell us your cruise → AI suggests everything → Refine → Book

User enters cruise details (ship, ports, dates). AI generates a complete shore plan with recommended activities per port, pre-scheduled within dock windows. User reviews, swaps activities, adjusts, then submits.

Pros

  • Minimum effort for user — just approve/tweak
  • Feels like a personal concierge
  • Strong differentiator vs competitors
  • Works great for first-time cruisers

Cons

  • Requires rich activity catalog per port
  • AI must know activity quality/ratings
  • Less control for detail-oriented users
  • Biggest engineering effort
